SPEAKER NOTES — Entropy-Guided Token Importance for PPO
========================================================

SLIDE 1: The Problem
--------------------
- Open with the PPO loss formula. Point out the 1/T — every token gets
  the same weight.
- Give the math example: "Let x = 3 + 4 = 7. Therefore the answer is
  \boxed{7}". The "7" is everything. "Let" and "Therefore" carry zero
  reward signal.
- This means the model is spending gradient budget on tokens that don't
  matter. Can we do better?

SLIDE 2: Prior Work — TI-DPO
-----------------------------
- TI-DPO showed that weighting tokens by importance improves DPO.
- They used gradient attribution, attention, etc. to score tokens.
- Key: these work for DPO because DPO has paired sequences (chosen vs
  rejected). You can identify which tokens are "different" between the
  good and bad response.
- This paper is our starting point.

SLIDE 3: Our First Step — Port to PPO
---------------------------------------
- Natural first idea: take TI-DPO's exact importance scores and use them
  as per-token weights in PPO. So w(t) = gradient norm, or attention
  received, or the hybrid score — the same signals from the paper.
- Explain what these scores mean: gradient norm measures how much the
  embedding at that position contributes to the loss. Attention measures
  how much other tokens "look at" this token. Hybrid smooths the gradient
  with a Gaussian prior.
- But in DPO these scores have a clear interpretation: they tell you
  which tokens differ between the chosen and rejected response. In PPO
  there's no rejected response — just one sequence and a scalar reward.
  So gradient/attention scores don't have the same meaning.
- Also: PPO runs for hundreds of gradient steps, so any systematic error
  in the weights accumulates. DPO is a single pass — bias can't compound.

SLIDE 4: Phase 1 Results
-------------------------
- Just a quick table showing the numbers. Don't dwell — the point is
  that direct porting doesn't work.
- Mention that the hybrid method does have lower KL, so it's not
  completely useless — it's just not optimizing what we care about
  (reward).
- Transition: we need signals that are native to PPO.

SLIDE 5: Entropy as Importance
-------------------------------
- THIS IS THE KEY SLIDE. Spend time here.
- Policy entropy at each token position: H(π(·|s_t)). This is how
  uncertain the model is about what comes next.
- High entropy = the model sees multiple plausible continuations.
  This means: (a) there's something to learn here, and (b) the PPO
  gradient carries useful information at this position.
- Low entropy = the model is confident. Maybe it's a function word,
  maybe it's already learned this pattern. Either way, the gradient
  at this token is mostly noise.
- We normalize so the mean weight is 1 (preserves the gradient scale)
  and use it in the PPO loss.
- Emphasize: this is FREE. Just softmax the existing logits. No extra
  forward/backward pass.

SLIDE 6: Entropy Intuition
---------------------------
- Walk through the table row by row.
- "Let" — entropy 0.2, model knows how to start. No point focusing here.
- "=" — entropy 1.4! The model has to compute the answer. There are many
  possible numbers it could output. This is the critical decision.
- "105" — entropy 1.2. The actual answer. High uncertainty because the
  model could output wrong numbers.
- Ask the audience: if you could only update the model on one token,
  which would you pick? Entropy identifies the answer automatically.

SLIDE 7: Early Results — Entropy Works... Then Doesn't
-------------------------------------------------------
- Show the table: entropy weighting is +10% at episode 100. Great!
- But at episode 150, baseline has overtaken it. What happened?
- The issue isn't that entropy is a bad signal. It's that we're
  applying the SAME weights throughout training, but the model changes.
- At step 1, the model is uncertain about math reasoning — entropy
  correctly identifies where to learn.
- At step 150, the model has learned a lot. The tokens that are now
  high-entropy might be high-entropy for different reasons — maybe the
  model is exploring alternatives, or maybe the entropy landscape
  shifted as the policy moved.
- The weights become stale. The small error per step accumulates over
  hundreds of steps into a large systematic distortion.

SLIDE 8: Why Fixed Weights Degrade
------------------------------------
- Formalize the intuition with the bias-variance decomposition.
- Bias = Cov(w, f). If the weights correlate with the gradient (they
  do — that's the whole point), the estimator is biased.
- Early: variance is high because the policy is still changing rapidly.
  Entropy weighting reduces variance → net win despite small bias.
- Late: the policy is more stable, so variance is naturally lower. But
  the accumulated bias from hundreds of steps with stale weights now
  dominates.
- NOTE: "early" and "late" here mean early/late in RL fine-tuning, not
  pretraining. The base model is already pretrained and capable. But
  the PPO process itself has dynamics — the policy changes most in the
  first few hundred updates, then stabilizes.

SLIDE 9: AITI — The Fix
-------------------------
- Simple fix: decay ε from 1 to 0. At the start, full entropy
  weighting. At the end, uniform PPO.
- The formula: w(t) = 1 + ε(step) × (H_norm(t) - 1). When ε=0,
  every weight is 1.
- Analogy to learning rate scheduling: just as you decrease the step
  size over training, you decrease the entropy weighting intensity.
  LR controls how far you step; AITI controls where you step.
- Hyperparameters: the number of decay steps. We tested 100 and 200.

SLIDE 10: AITI Results
-----------------------
- Show the table. AITI-Entropy Pareto-dominates PPO baseline: higher
  reward AND 2.5x lower KL divergence.
- Fixed entropy (no decay) is dominated — confirms the degradation story.
- The KL result is interesting: AITI achieves lower KL than baseline,
  meaning the policy stays closer to the reference while getting better
  reward. The entropy weighting focuses updates efficiently.
- TI-DPO hybrid is also dominated, confirming Phase 1 findings.

SLIDE 11: MOAI — Can We Do Better Than Scheduled Decay?
---------------------------------------------------------
- AITI requires choosing a decay schedule. What if we compute the
  optimal ε at each step?
- MSE of the gradient estimator = bias² + variance. We want the gradient
  that's closest to the true gradient in mean squared error.
- Why MSE? Because the PPO update is θ ← θ + α·g. A lower-MSE estimate
  of g gives better updates on average. MSE = bias² + variance is the
  standard decomposition of estimation error.
- The closed form: ε* = clip(-C/2ρ, 0, ε_prev). The statistics C and ρ
  are estimated via EMA from the training data.
- Monotone constraint: ε can only decrease. This prevents oscillation.

SLIDE 12: MOAI Finding — Partial Weighting
--------------------------------------------
- Interesting result: MOAI settles at ε ≈ 0.3-0.5, not zero.
- This seems to contradict AITI (which decays to 0). Explain why:
  - AITI goes to 0 to eliminate ALL bias. This is safe but conservative.
  - MOAI says: at ε=0.3, the remaining bias² is negligible, but you're
    still getting meaningful variance reduction from the entropy weights.
  - Going from ε=0.3 to ε=0 removes very little bias but gives up
    real variance reduction. Not worth it.
- So MOAI finds a more nuanced answer: keep some entropy weighting
  permanently, just not as much as at the start.
- Show the full table: MOAI gets highest reward, AITI gets best KL.
  Both Pareto-dominate PPO. Different operating points on the frontier.

SLIDE 13: Implementation
--------------------------
- Don't skip this slide — these bugs were real and devastating.
- fp16 value head: the gradient norm was 4.2 on the first step. In fp16
  this overflows to NaN. Every subsequent step was silently skipped.
  The model appeared to train but learned nothing.
- fp16 log-softmax: the KL between policy and reference was literally
  0.0000 for the entire run. The PPO ratio was exactly 1.0. No policy
  gradient at all.
- Chat template: Qwen instruct models expect <|im_start|>user\n...<|im_end|>
  format. Without it, the model doesn't enter "instruction following" mode
  and accuracy drops 12%.
- Small training pool: with 500 questions and batch_size=4, the model
  sees most questions multiple times → memorization, not learning.

SLIDE 14: GSM8K Results
-------------------------
- All methods improve 8% over base model. Good — PPO works.
- But differences between methods are <1%. The task is too easy for the
  1.5B model (base already at 55%) to differentiate methods.
- The batch size finding is more interesting: at bs=1, entropy methods
  give +4-5% over baseline. This confirms the variance reduction story.
  Small batches = high variance = entropy weighting helps.
- If you're training with small batches (common in RLHF due to memory),
  entropy weighting is essentially free improvement.

SLIDE 15: MATH-500 Results
----------------------------
- Harder task. Base model gets ~30%. PPO only improves by ~1.6%.
- With so little total PPO improvement, entropy methods can't add much.
- BUT: in training accuracy, entropy methods are +4% ahead (28% vs 24%).
  This means the method IS helping with learning — it just doesn't show
  in the test accuracy because the total effect is too small.

SLIDE 16: Takeaways
---------------------
- Four points. Rehearse these — they're the main message.
  1. Entropy = right signal (content-aware, free, identifies decision points)
  2. Fixed weighting degrades (bias accumulation in online RL)
  3. AITI/MOAI fix it (adaptive decay or per-step optimization)
  4. Most valuable in high-variance settings (small batch, hard tasks)
- The framing: this is variance reduction for the policy gradient.

SLIDE 17: Future Work
-----------------------
- Scaling is the biggest open question. We need tasks where PPO makes a
  big difference so we can measure whether entropy weighting makes it
  even bigger.
- Code generation is a natural fit — variable names have low entropy
  (determined by context) while logic has high entropy (many valid
  implementations).
- GRPO integration would be interesting — GRPO is gaining popularity
  as a PPO alternative, and it has the same per-token loss structure.
- End on the core insight: variance reduction. Simple, free, additive.
